# Common Language

This page defines the shared language used across the public GitHub ecosystem.

The goal is to keep the portfolio precise. These terms should mean the same thing across repos, READMEs, docs, playbooks, and handoff material.

## Operating system

The standards, workflows, roles, decision logic, QA checks, documentation, and cadence that make growth work easier to manage.

In this ecosystem, an operating system is not software alone. It is the structure underneath repeated work.

## Source of truth

The file, repo, table, document, or system that owns the canonical version of a definition, value, rule, or relationship.

A source of truth should reduce debate. If two places disagree, the source of truth wins until it is intentionally updated.

## Decision infrastructure

The reporting, context, metrics, meeting rhythm, templates, and escalation logic that help leaders make better decisions faster.

Decision infrastructure turns raw information into tradeoffs, risks, recommendations, and next actions.

## Intake layer

The place where raw inputs are gathered, organized, and made usable.

Examples include brand rules, voice notes, code references, Figma notes, assets, web examples, prompts, campaign inputs, stakeholder context, or reporting exports.

The intake layer should not pretend to be the final system. Its job is to collect and structure inputs cleanly.

## Implementation layer

The place where structured inputs become reusable execution assets.

Examples include design tokens, component specs, CSS variables, scripts, playbooks, templates, validation rules, and handoff guidance.

The implementation layer should be usable by humans and tools without requiring a long verbal explanation.

## Governance

The rules, ownership, review standards, naming conventions, and decision rights that keep a system from drifting.

Good governance should improve performance and clarity. It should not add bureaucracy for its own sake.

## Repeatable execution

A workflow that can be run again by another person or tool with clear inputs, steps, review criteria, and expected outputs.

Repeatable execution is the opposite of heroics. The work should not depend on one person holding every detail in their head.

## Signal

The information that changes the decision.

Signal is not the same as data volume. A report can contain a lot of data and very little signal. A useful system separates what matters from what merely looks urgent.

## Diagnostic lens

A structured way to look at a problem before prescribing a fix.

Examples include demand reality, measurement integrity, media allocation, agency model, team ownership, or leadership cadence.

A diagnostic lens prevents premature conclusions.

## Handoff

The structured transfer of context, standards, assets, and expectations from one person, team, repo, or tool to another.

A good handoff makes the next action clearer and reduces re-explanation.

## AI-assisted workflow

A workflow where AI helps with synthesis, drafting, QA, classification, monitoring, or transformation, while human judgment remains responsible for review and decisions.

AI assistance should make the system more consistent, not less accountable.

## Public-safe

Suitable for public portfolio use because it avoids private data, confidential details, unsupported claims, and sensitive operational specifics.

Public-safe material can still be useful and specific, but it should not expose private employer, client, candidate, vendor, or customer information.

## Proof discipline

The habit of ensuring public claims are supported by source material, experience, artifacts, or clearly stated assumptions.

Proof discipline prevents the portfolio from sounding more certain than the underlying evidence supports.

## System drift

The gradual divergence between what a system says should happen and what actually happens in practice.

Examples include naming conventions eroding, tokens changing without documentation, reports using inconsistent definitions, or repos repeating conflicting descriptions.

## Canonical map

The single place that explains how the repos relate to each other.

For this GitHub ecosystem, the canonical map is `docs/ecosystem-map.md` inside Growth Architecture OS. Other repos should link to it rather than restating the relationship logic.
