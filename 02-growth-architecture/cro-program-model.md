# CRO Program Model

## Principle

CRO is not button testing. CRO is the operating discipline of making conversion easier to understand, improve, and scale.

A useful CRO program connects research, funnel friction, proof, UX, measurement, test design, decision cadence, and rollout discipline.

## Five workstreams

### 1. Foundation

The basic layer that makes testing trustworthy:

- Analytics and event tracking
- Funnel definitions
- Form visibility
- Landing page inventory
- Site speed and mobile checks
- Source-of-truth reporting

### 2. Test queue

A prioritized backlog of testable hypotheses.

Each test should define:

- Hypothesis
- Location
- Primary metric
- Guardrail metric
- Expected impact
- Effort
- Confidence
- Minimum sample requirement
- Decision rule

### 3. Rollout model

Winners need a path beyond the page where they were tested.

Define:

- What ships immediately
- What requires design or development
- What rolls into other brands or states
- What becomes a standard pattern
- What gets retired

### 4. Measurement cadence

CRO needs a rhythm:

- Weekly stand-up
- Biweekly readout
- Monthly roadmap refresh
- Quarterly strategic reset

The point is not to keep tests running. The point is to make better conversion decisions faster.

### 5. Operating model

Clarify who owns:

- Roadmap priority
- Hypothesis design
- Build and QA
- Copy and proof
- Visual design
- Tracking
- Readout and decision

## What good looks like

A mature CRO program should produce:

- Fewer opinion debates
- Clearer test rationale
- Better funnel visibility
- More reusable learnings
- Faster rollout of winners
- A shared understanding of what blocks conversion

## Standard

Measure before you move. Then sequence the work by dependency, not enthusiasm.
